# Security Rules

> Establishes the defensive security posture for the repository.

---

## Purpose

Security is paramount. This document defines the strict policies to ensure that the repository remains secure, free of credentials, and resilient against vulnerabilities. All agents must enforce these rules in every action they perform.

---

## Defensive-Only Policy

All security-related actions and content generated by agents must be strictly **defensive**.

- **Allowed:** Defensive audits, secure coding practices, dependency vulnerability review, secrets detection, hardening configurations, safe architecture design, incident prevention mechanisms.
- **Prohibited:** Offensive exploitation instructions, credential harvesting scripts, malware generation, persistence mechanisms, evasion techniques, unauthorized access instructions, or real attack playbooks.

---

## Secrets and Credentials Policy

- **Never hardcode secrets:** API keys, passwords, private keys, database credentials, or tokens must never be written into source code or documentation.
- **Secrets detection:** Agents must validate their own output to ensure no secrets are accidentally included.
- **Placeholder values:** Use clear placeholder syntax (e.g., `YOUR_API_KEY`, `<secret-token>`) when examples require credentials.

---

## Environment File Policy

- `*.env` files containing actual secrets must be added to `.gitignore`.
- Only `.env.example` or `.env.template` files containing dummy placeholder data may be committed to the repository.

---

## Dependency Review Policy

- Agents must avoid introducing unnecessary dependencies.
- When adding a dependency, agents should verify it is from a trusted source and not a known malicious package (e.g., typosquatting).

---

## Authentication and Authorization Review

- Agents implementing authentication must follow standard, secure protocols (e.g., OAuth 2.0, JWT with proper signing and expiration).
- Authorization checks must be explicit and enforced at the server level.

---

## Secure Development Practices

### Input Validation
All user-provided data must be strictly validated, sanitized, and type-checked before processing to prevent injection attacks (SQLi, XSS, etc.).

### Secure Logging
- Do not log sensitive information (passwords, session tokens, PII).
- Ensure error messages exposed to the user do not reveal internal system details or stack traces.

### Privacy and Personal Data
- Do not collect, store, or process personal data unnecessarily.
- When generating mock data for tests, use clearly synthetic data (e.g., `testuser@example.com`).

---

## Production Readiness

Before suggesting code for production, agents must consider:
- Rate limiting to prevent abuse.
- Secure transport (HTTPS/TLS).
- Proper configuration of CORS and security headers (e.g., CSP).
