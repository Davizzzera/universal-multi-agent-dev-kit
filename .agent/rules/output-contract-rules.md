# Output Contract Rules

> Defines the standardized structure for how agents must report their work upon completion.

---

## Purpose

Unstructured agent responses are difficult for the Universal Orchestrator and other agents to parse. Output contracts enforce a predictable, structured response format that guarantees all necessary information (work done, validations, blockers) is communicated clearly.

---

## Why Output Contracts Matter

- **Machine Readability:** The orchestrator can easily extract status and next steps.
- **Accountability:** Agents are forced to explicitly state what they changed and how they validated it.
- **Context Preservation:** Handoffs become seamless when the output of Agent A perfectly matches the expected input of Agent B.

---

## Standard Agent Output Structure

Every agent completing a task MUST return a final response adhering to this structure.
**Crucially, every implementation, debugging, refactoring, release, or validation task MUST include an Execution Trace in the final response.**
Complex tasks should include an Orchestration Plan before implementation.

### 1. Execution Trace (Mandatory for implementation tasks)
- **Workflow used** and **Pack used**.
- **Execution Mode**: Explicitly state if subagents were tool-visible or orchestrated roles.
- **Agents**: Distinguish between selected, actually used, and recommended agents.
- **Skills**: Distinguish between selected and actually used skills.
- **Files**: List files analyzed, created, and modified.

### 2. Summary
A concise description of the task performed.

### 2. Files Changed
A clear list of files that were created, modified, or deleted.
- Must use absolute or repository-root-relative paths.
- Must explicitly state if *no* files were changed.

### 3. Validation Performed
Explicit proof that the Validation-First rule was followed.
- What tests/checks were run.
- The outcome (Pass/Fail).
- **Evidence Requirements:** Include brief snippets of successful command output, or a checklist if manual.

### 4. Risks or Limitations
Honest reporting of potential edge cases, technical debt incurred, or security considerations.
- Must state "None identified" if none exist.

### 5. Suggested Next Step
A clear recommendation for what should happen next.
- e.g., "Handoff to QA-Tester to verify changes," "Proceed to step 3 of workflow," or "Wait for user feedback."

---

## Standard Skill Output Structure

Skills are functions executed by agents. Their output must be predictable:

- **Status:** Success / Failure
- **Data:** The raw output generated by the skill (e.g., file contents, command output).
- **Error Details:** If Failure, a clear description of why it failed.

---

## Standard Workflow Output Structure

When a workflow completes, the orchestrator outputs:

- **Workflow Status:** Completed / Blocked / Failed
- **Steps Executed:** List of steps successfully run.
- **Final Validation Status:** Aggregated validation results.
- **Delivery Artifacts:** The final state delivered to the user.

---

## Diff Summary Requirements

If an agent modifies files, it should provide a high-level summary of *what* changed logically, not just a raw git diff.
- Example: "Added `login` endpoint to `routes.ts` and exported it."

---

## Validation Summary Requirements

The validation summary must explicitly link the validation performed to the changes made.
- Example: "Ran `npm test src/auth.test.ts`. All 4 tests passed. Evidence: `PASS src/auth.test.ts`."

---

## Blocker Reporting

If an agent cannot complete its output contract due to a blocker, it must output an **Error Contract**:

- **Summary:** What task failed.
- **Blocker:** Exact description of what prevents progress (e.g., "Missing API Key", "Syntax error in dependency").
- **Attempted Solutions:** What the agent tried to do to fix it.
- **Escalation Target:** Who needs to resolve this (User, Orchestrator, DevOps Agent).
